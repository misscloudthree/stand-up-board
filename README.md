# 站會看板 Standup Board

一個部署在 GitHub Pages 上、免自架後端的站會追蹤工具。整個網站是純靜態網頁，**所有資料都直接寫成這個 repo 的 GitHub Issues**（用隱藏的資料區塊存結構化欄位、用 labels 做分類查詢、用 Issue 留言功能做討論串）。只要大家都指向同一個 repo，資料就是共用的。

## 功能總覽

網站上方分成 4 個頁籤：

| 頁籤 | 功能 |
|---|---|
| **Backlog 追蹤** | 記錄站會議題、類型、負責人、deadline、提出人/日期，用看板方式管理狀態（待處理/進行中/已完成/作廢），並可留言討論。「類型」是可自訂下拉輸入框，會帶入目前已用過的類型，也可以直接輸入新分類。工具列可依狀態/類型/負責人篩選、依 Deadline 排序。 |
| **每日測試追蹤** | 多人可各自輸入 FSD 模組的每日追蹤數字，並自動加總成當日 Summary；下方「趨勢圖表」可切換依週/依月，用長條圖看各追蹤項目的變化。 |
| **交付測試追蹤** | 依模組(FSD)分組總覽每個功能項目的交付/FT/SIT 測試狀態，含交付時程區間、逾期/已交付/部分交付統計、FT/SIT 狀態分布、SIT TC 總數與已測通數、待補 SIT TC 提示；可設定各交付階段的 FT/SIT 預計完成日與每個功能項目的基本測試天數，系統會自動回推預計測試時程並標示「測試時程不足」的風險項目；也可下載 Excel 範本、上傳填好的 Excel 批次新增/更新模組與功能項目資料。 |
| **看板設定項目** | 集中管理「FSD 模組管理」與「每日測試追蹤項目設定」，頁面上會說明每項設定會影響哪些頁籤/功能。 |

## 部署步驟（全新部署）

### 1. 建立一個 GitHub repo

到 GitHub 建立一個新的 repository（例如叫 `standup-board`）。

> **公開 vs 私有**：GitHub Pages 若要免費使用，repo 必須是 **Public**（私有 repo 的 Pages 功能需要 GitHub Pro / Team / Enterprise 帳號）。因為資料是寫在這個 repo 的 Issues 裡，選 Public 代表任何人都能在 GitHub 上看到這些 Issues 內容。如果你們的站會內容較敏感，建議：
> - 改用 GitHub 內部/組織帳號的私有 repo + 付費方案，或
> - 只把「網頁靜態檔案」放在一個 public repo 做 Pages，但把 Issues 資料放在另一個 private repo（把 `config.js` 的 `GITHUB_REPO` 指向那個 private repo；team 成員的 Token 需要有該 private repo 的存取權限即可，Pages 網址仍是 public repo 的）。

### 2. 修改 `config.js`

打開 `config.js`，把兩個值改成你剛建立的 repo 資訊：

```js
window.APP_CONFIG = {
  GITHUB_OWNER: "你的GitHub帳號或組織名稱",
  GITHUB_REPO: "standup-board"
};
```

### 3. 把檔案推上 GitHub

在這個資料夾內執行：

```bash
git init
git add .
git commit -m "init standup board"
git branch -M main
git remote add origin https://github.com/<你的帳號>/<repo名稱>.git
git push -u origin main
```

### 4. 開啟 GitHub Pages

repo 頁面 → **Settings** → **Pages** → Source 選擇 **Deploy from a branch** → Branch 選 `main` / `/ (root)` → Save。

等 1-2 分鐘後，GitHub 會給你一個網址，格式通常是：

```
https://<你的帳號>.github.io/<repo名稱>/
```

這就是你要分享給團隊的共用連結。

### 5. 每位使用者設定自己的 Token

因為要寫入 GitHub Issues，每個使用者（包含你自己）都需要一組屬於自己的 **Personal Access Token**：

1. GitHub 右上角頭像 → **Settings** → 左側最下方 **Developer settings** → **Personal access tokens** → **Fine-grained tokens** → **Generate new token**
2. **Repository access** 選 **Only select repositories**，選剛剛那個 repo
3. **Permissions** → **Repository permissions** → 找到 **Issues**，設為 **Read and write**
4. 產生後複製 token（`github_pat_...` 開頭），**它只會顯示一次**
5. 打開站會看板網站 → 右上角「⚙ 設定」→ 貼上 Token，填上自己的顯示名稱 → 儲存

Token 只會存在**該使用者自己瀏覽器**的 localStorage，不會被送到 GitHub 以外的任何地方，也不會被其他使用者看到。每個人要各自貼自己的 Token（不要共用同一組，方便之後個別撤銷）。

## 客製化你的追蹤內容（給全新部署者）

如果你不是要照搬原團隊的追蹤項目，而是想套用到自己專案，有以下幾種做法，依「要不要改程式」分開說明。

### 1. 不用改程式：直接在網頁上設定模組與每日追蹤欄位

「Backlog 追蹤」和「每日測試追蹤」的追蹤內容本來就是可設定的，不用碰程式碼：

- 到「**看板設定項目**」頁籤 →「**FSD 模組管理**」新增/刪除你要追蹤的模組（例如把 FSD 模組換成你的專案模組、team、或功能區塊名稱），可同時填負責人。
- 同頁「**每日測試追蹤項目設定**」可新增/刪除/改名要追蹤的每日欄位（例如把「開單量／複測數量／複測通過量／Pre-UAT 處理量」換成你自己想追的指標，如「新增 bug 數／修復數／待測項目」等），至少需保留一項欄位。存好後「每日測試追蹤」表格欄位與趨勢圖表會立刻套用新設定。

這兩項設定會影響「每日測試追蹤」表格欄位與加總、趨勢圖表項目，以及「交付測試追蹤」的模組分組依據，頁面上有列說明。

### 2. 想改「交付測試追蹤」的表格格式：用 AI vibe coding 重新套用

「交付測試追蹤」的 Excel 欄位（28 欄）目前是寫死比照團隊原本的《SIT測試管理五大表格》，不像上面兩項是網頁可設定的，如果你的專案追蹤表格式完全不同（欄位、驗證規則、卡片統計方式都不一樣），建議直接請 AI（例如 Claude Code）依你的實際 Excel 表格重新套用，程式碼都集中在 `app.js` 幾個地方，很適合請 AI 直接改：

- `TEST_TEMPLATE_COLUMNS`（約第 45 行）：Excel 範本的欄位清單與表頭文字，改這裡就會連動「下載範本」「上傳解析」「卡片明細顯示」。
- `REQUIRED_TEST_FIELD_KEYS` / `ROW_REQUIRED_KEYS`（約第 78-83 行）：哪些欄位表頭必須存在、哪些欄位每列不可空白。
- `TEST_DATE_FIELD_KEYS`（約第 84 行）：哪些欄位要當日期格式解析（`parseExcelDateValue`，約第 1562 行）。
- `downloadTemplate()`（約第 1545 行）：產生 Excel 範本。
- `onUploadExcelChange()`（約第 1729 行）：上傳解析、驗證、寫回 Issue 的邏輯。
- 模組卡片統計、風險判斷（測試時程回推）等顯示邏輯集中在同一支檔案的「Testing overview」區塊（約第 1234 行之後）。

**建議跟 AI 說的內容（可直接照抄調整）：**

> 我要把「交付測試追蹤」的 Excel 範本欄位換成我自己專案的表格格式，欄位是：[貼上你的欄位清單，標明哪些是必填、哪些是日期]。請幫我改 `app.js` 裡的 `TEST_TEMPLATE_COLUMNS`、`REQUIRED_TEST_FIELD_KEYS`、`ROW_REQUIRED_KEYS`、`TEST_DATE_FIELD_KEYS`，以及對應的卡片統計顯示邏輯，讓下載的範本、上傳驗證訊息、模組卡片統計都改用新欄位。

**改完後預期要看到的結果（拿來驗收）：**

- 「下載 Excel 範本」下載下來的表頭，要跟你新定義的欄位一致。
- 上傳你自己整理好的 Excel，能被正確解析、新建/更新模組與功能項目；刻意留空必填欄位或填錯日期格式時，上傳前會列出「第幾列、哪個欄位」的錯誤清單並整批中止（不寫入部分資料）。
- 模組卡片上的統計數字（例如逾期/已交付、狀態分布）改用你新欄位的邏輯計算，不再顯示舊的「FT/SIT」字樣，除非你就是要保留類似邏輯。

### 3. 不想改寫程式：直接用範例 Excel 規劃專案追蹤

如果暫時不想動程式碼，也可以到「交付測試追蹤」頁籤按「**下載 Excel 範本**」，把這份範本當成純粹的離線 Excel 規劃表使用（自己填、自己存、不上傳回網站），先用熟悉的 Excel 規劃專案追蹤內容，之後有需要再考慮要不要照上面第 2 點請 AI 改成網站真正支援的欄位。

### 4. 不想用 GitHub Issue 當資料庫：換成 Firebase 等其他後端

這個專案目前的「資料庫」就是 GitHub Issues，所有讀寫都集中在 `app.js` 呼叫 GitHub REST API 的部分（`ghFetch`、各個 `load*`/`save*` 函式）。如果你不想讓資料存在 GitHub Issues（例如想要即時同步、想要更細緻的權限控管、資料量會超過 Issue 的限制等），可以考慮換成 Firebase（Firestore + Authentication）、Supabase 等有現成免費額度的後端服務，但要注意：

- 這是**架構層級的改動**，不是改幾個設定值就好，等於要把 `app.js` 裡讀寫 GitHub Issues 的部分整個換成呼叫新後端的 SDK/API，包含 Backlog、模組、每日追蹤、交付測試追蹤全部四大類資料。
- 建議一樣用 AI vibe coding 進行，但要先想清楚：資料結構要沿用目前的（`data` JSON + labels）還是重新設計、要不要保留留言功能（Firebase 需要另外設計 comments collection）、身分驗證要用 Firebase Auth 還是沿用現在「自己輸入顯示名稱」的簡易方式。
- 換掉後端後，本 README 中「Token 申請」「GitHub Issues 當資料庫」相關段落都需要對應改寫，避免誤導後續使用者。

若只是單純不想用 Public repo 曝光 Issues 內容，且不想大改架構，可先參考上面「部署步驟」第 1 步的私有 repo 作法，通常比換後端簡單很多。

## 團隊共用同一個網站怎麼用（給 Coworker）

團隊成員要一起用同一個網址追蹤同一份資料時，請注意以下幾點：

### 1. 每人都要申請自己的 Token

步驟同上方「部署步驟」第 5 步：每個人各自到 GitHub 申請一組 **Fine-grained personal access token**，權限只需要該 repo 的 **Issues：Read and write**，然後在網站右上角「⚙ 設定」貼上自己的 Token。**不要共用同一組 Token**，一來方便日後個別撤銷，二來 Token 不影響「誰是誰」的辨識（辨識靠的是下面第 2 點的顯示名稱，不是 Token）。

### 2. 顯示名稱才是身分辨識依據，換 Token／換裝置時名稱要打一模一樣

這是最容易踩雷的地方：這個網站**沒有真正的登入系統**，「⚙ 設定」裡的**顯示名稱**只是存在你瀏覽器 localStorage 的一段文字，用來當「提出人」「回報人」的預設值，並不是綁定你的 GitHub 帳號。也就是說：

- 換一台電腦、換瀏覽器、清過 localStorage、或是重新申請/更新 Token 之後，**只要顯示名稱重新打的文字跟之前不完全一樣（大小寫、全形半形、空格都算不一樣）**，系統就會把你當成「另一個人」，之前用篩選負責人/查詢自己名下項目時可能就對不起來。
- 建議團隊先統一講好每個人的顯示名稱要怎麼打（例如統一用英文名或統一用中文全名），設定好之後盡量不要隨意更動，換裝置時務必照抄一模一樣的字。
- Backlog 卡片上的「負責人」欄位也是純文字手動輸入，不會自動抓你的顯示名稱，篩選負責人時同樣要注意大家打的名字要一致。

### 3. 多人同時編輯同一份每日追蹤資料時會互相覆蓋

「每日測試追蹤」的每一列（模組＋日期）是**同一個 GitHub Issue**，不是每人各自一份，所以若同一個模組、同一天有兩人先後按「儲存」，**後存的人會整列覆蓋掉先存的人**（不是逐欄合併）。建議團隊約定每個模組每天由固定一人回報，或至少先按「重新整理」拿到最新資料再填寫、儲存。

## 各頁籤使用方式

- **Backlog 追蹤**：點「＋ 新增 Backlog」建立議題；點卡片可修改類型/負責人/deadline/狀態，並在下方留言記錄討論或處理進度；deadline 在兩天內或已逾期的卡片會自動標色提醒。工具列可勾選要顯示哪幾種狀態（預設只勾待處理/進行中）、依類型或負責人篩選、切換依 Deadline 近到遠或遠到近排序。
- **每日測試追蹤**：先到「看板設定項目」新增 FSD 模組與負責人、設定要追蹤的欄位（至少需保留一項）；回到本頁籤選擇日期後，每個模組一列可輸入當日數字，按「儲存」寫回 GitHub；下方 Summary 會自動加總目前畫面上所有模組的數字。切換日期或有其他人更新資料後，按「重新整理」重新拉取最新資料（不是即時同步）。再下方「趨勢圖表」會抓全部歷史每日紀錄，依週（近 8 週）或依月（近 6 個月）加總畫成長條圖，新增/移除追蹤項目或剛儲存的當日數字都會立即反映在圖表上。
- **看板設定項目**：「FSD 模組 / 負責人管理」跟項目一的模組維護共用同一份資料；「每日測試追蹤項目設定」控制的是每日測試追蹤表格的欄位與加總、趨勢圖表要畫哪些項目。這兩塊都會影響其他頁籤的顯示，頁面上方有條列說明。
- **交付測試追蹤**：
  - 點「下載 Excel 範本」取得欄位比照團隊既有《SIT測試管理五大表格》第一工作表的空白範本（FSD、FSD名稱、第一層/第二層、測試情境ID、預計交付時程、交付狀態、FT/SIT測試狀態、TC數量等 28 欄）。
  - 填好後點「上傳 Excel 更新」：系統會逐列檢查必要欄位與日期格式，若有問題會列出「第幾列、哪個欄位」的錯誤清單，**整批中止匯入**，不會寫入部分資料；全部通過才會寫入。上傳時若出現 Excel 裡沒建過的 FSD，會自動在「模組管理」新增對應模組。同一個「測試情境ID」重複上傳視為更新（不會產生重複項目），狀態欄位有變動時舊值會保留一份簡短的變更紀錄。
  - 匯入後可在「測試階段設定」摺疊區塊，針對資料中出現過的每個「交付階段」填 FT/SIT 預計完成日，並設定全域「基本測試天數」預設值；系統會用「該功能交付時程」+「基本測試天數」回推預計測試時程，並在可測試期間不足時標示風險。「交付階段」空白的項目不參與回推，畫面上會標示「未設定交付階段」。
  - 模組卡片依 FSD 分組，顯示交付時程區間、交付狀態統計（逾期未交付／已交付／部分交付）、FT/SIT 狀態分布、SIT TC 總數與已測通數；SIT 非「不需測試」但 TC 數量空白時會提示「需要補上 SIT TC」。點卡片可展開查看每個功能項目的明細與個別覆寫「基本測試天數」。

## 已知限制

- 沒有即時同步，多人同時看畫面時需要各自按「重新整理」才會看到別人剛存的資料；「每日測試追蹤」同一模組同一天的資料是單一 Issue，後存的人會整列覆蓋先存的人（見上方「團隊共用」第 3 點）。
- 目前一次讀取上限抓 100 筆（Backlog items、模組、追蹤項目等清單），對一般團隊用量足夠；若累積超過 100 筆 backlog，較舊的項目可能不會顯示在清單中，需要另外加分頁功能。例外是「每日測試追蹤」的趨勢圖表：它需要撈全部歷史每日紀錄，這個查詢本身有做真正的分頁（撈到不滿 100 筆才停止，上限約 2000 筆），所以資料量持續累積也不會漏抓。
- Token 存在瀏覽器 localStorage：若使用共用電腦，建議使用完在瀏覽器清除資料，或使用權限範圍最小的 fine-grained token。顯示名稱同樣只存在瀏覽器本機，換裝置需要手動重新輸入且要打一模一樣（見上方「團隊共用」第 2 點）。
- 受 GitHub API 速率限制（已登入 Token 每小時 5000 次請求），一般團隊規模不會碰到上限。
- 「交付測試追蹤」的測試情境資料量可能上千筆，若比照其他功能「一筆一個 issue」會嚴重超過速率限制，因此改成**每個模組(FSD)合併成一個 issue**（body 存整批 JSON）。代價是單一測試情境沒有自己的 GitHub issue、無法個別留言；單一模組底下的測試情境筆數建議不要超過數百筆（issue body 上限 65536 字元）。
- 「交付測試追蹤」的 Excel 欄位目前是寫死在程式碼裡（`app.js` 的 `TEST_TEMPLATE_COLUMNS` 等常數），不像模組/每日追蹤欄位可以直接在網頁上設定；要換成自己專案的表格格式需要改程式碼，作法見上方「客製化你的追蹤內容」第 2 點。
- Excel 讀寫透過外部 CDN（cdn.sheetjs.com）載入函式庫，需要網路連線才能下載/上傳範本；離線環境無法使用「交付測試追蹤」的 Excel 功能。
